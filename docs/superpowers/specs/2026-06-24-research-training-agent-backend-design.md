# 论文研究能力训练型 Agent：课程演示版后端设计

日期：2026-06-24

状态：待用户审阅

依据：《产品说明.docx》及后续需求讨论

## 1. 项目目标

本项目面向研究生新手，提供论文研究能力训练，而不是论文代写。课程演示版需要形成可自由体验的完整产品闭环：

1. 从模糊研究兴趣出发发现真实文献；
2. 通过引导精读或快速分析理解论文；
3. 将文献所得转化为选题、研究设计和论文框架；
4. 将对话过程沉淀为可持续使用的研究项目档案。

产品采用“对话优先、项目自动沉淀”的交互逻辑。用户打开产品即可提问；首次产生有效研究内容时，系统自动创建未命名项目，无需登录、注册或预先填写项目表单。

## 2. 适用范围

首版重点服务 arXiv 覆盖较好的研究领域，包括但不限于：

- 管理科学；
- 工业工程；
- 物流工程与管理；
- 人工智能；
- 计算机科学；
- 软件工程；
- 相关交叉领域。

产品界面、操作引导和分析结果以中文为主，同时支持：

- 中英文检索词生成；
- 英文 arXiv 文献检索；
- 中英文 PDF 解析与精读；
- 中英文原文证据展示。

## 3. 课程版边界

### 3.1 首版包含

- 单机 Windows 本地运行；
- 单一聊天入口和自动模式路由；
- 普通研究问答；
- arXiv 真实文献检索；
- arXiv 论文导入；
- 本地 PDF 上传；
- PDF 文本解析和 OCR 兜底；
- 苏格拉底式引导精读；
- 基于证据的快速分析；
- 最多三篇论文对比；
- 选题、研究设计和框架诊断；
- 研究项目、对话和成果持久化；
- Markdown 编辑与导出；
- Qwen 流式回复；
- 本地设置和运行日志。

### 3.2 首版不包含

- 登录、注册、多用户和权限管理；
- 云端同步和多人协作；
- Redis、Celery、MinIO 等独立基础设施；
- 完全自主的多 Agent 协作；
- FAISS 或独立向量数据库；
- Zotero、BibTeX、RIS 和 EndNote 导入；
- 复杂引用网络、聚类和被引量分析；
- Word、PDF 等复杂格式导出；
- 查重、自动排版和完整论文代写；
- 对复杂公式、跨页表格和双栏版式作完全准确识别的承诺；
- 预置示例项目或示例论文。

## 4. 核心设计原则

### 4.1 模型负责推理，程序负责约束

Qwen 负责意图理解、动态追问、论文分析和研究诊断。程序负责：

- 流程路由；
- 状态保存；
- 工具调用；
- 文件限制；
- 文献真实性；
- 证据和页码绑定；
- 结构化结果校验；
- 错误处理与降级。

### 4.2 受控工作流，而非自由多 Agent

系统可以使用不同角色提示词完成检索、精读、核验和诊断，但不允许多个 Agent 自主拆解任务并自由协作。LangGraph 负责明确的流程和状态边界，以提高演示稳定性、可解释性和可调试性。

### 4.3 动态诊断，而非固定提问树

量化实证、质性研究、案例研究、文献综述、工程实验等类型只作为可组合的诊断检查清单。Qwen 可以识别交叉研究类型，并根据用户已有信息动态决定：

- 当前最重要的信息缺口；
- 下一步应提出的问题；
- 是否需要进一步追问；
- 何时可以形成诊断结果。

LangGraph 不微观规定每个问题的固定顺序。

### 4.4 关键判断必须绑定证据

文献总结、理论识别、方法判断和论文对比应尽可能绑定：

- 原文片段；
- PDF 页码；
- 文本块编号；
- 是否来自 OCR；
- 必要时的页面区域或截图。

证据不足时，系统必须标记“无法从当前论文确认”或“需要人工核实”，不得虚构页码和依据。

## 6. 技术选型

| 层级 | 技术 | 用途 |
|---|---|---|
| API | FastAPI | 接口、上传、SSE、后台任务 |
| 数据模型 | Pydantic | 请求校验和模型结构化输出校验 |
| ORM | SQLAlchemy 或 SQLModel | SQLite 数据访问和未来迁移 |
| 工作流 | LangGraph | 路由、子图、状态和流程恢复 |
| 集成层 | LangChain | arXiv、模型及文档组件封装 |
| 模型 SDK | OpenAI Python SDK | 调用百炼 OpenAI 兼容接口 |
| 主数据库 | SQLite | 项目、消息、论文、成果和任务 |
| 全文检索 | SQLite FTS5 | 论文证据关键词检索 |
| PDF 解析 | PyMuPDF | 文本、页码、文本块和坐标提取 |
| OCR | Tesseract 5.5 | 扫描 PDF 中英文识别 |
| 导出 | Markdown | 文献卡片、诊断和对比表导出 |

Python 后续使用本机 `py39232` 虚拟环境。Tesseract 安装路径通过设置或环境变量配置；当前开发环境已确认包含 `chi_sim`、`chi_sim_vert`、`eng` 和 `osd` 语言包。

## 7. 模型服务设计

### 7.1 基本配置

- 平台：阿里云百炼；
- 调用方式：OpenAI 兼容接口；
- 模型 ID：`qwen3.7-plus`；
- 模型名、`base_url` 和 API Key 均可配置；
- 回复采用流式输出；
- 多模态能力作为增强功能，不是核心流程的必要条件。

### 7.2 多模态使用范围

仅在以下场景按需发送页面截图或局部图像：

- 复杂表格；
- 数学公式；
- 图片型图表；
- 纯文本解析无法理解的页面区域。

视觉调用失败时，系统回退到正文文本分析，并提醒用户人工核验。

### 7.3 结构化输出

文献卡片、诊断报告、论文对比和项目档案更新均使用固定 Pydantic Schema：

```text
Qwen JSON 输出
→ Pydantic 校验
→ 失败时自动修复一次
→ 再次失败时保留普通文本并返回可理解错误
```

### 7.4 调用保护

- 只发送当前任务所需证据片段；
- 限制单次输入长度；
- 限制最大输出 token；
- 网络或临时模型错误最多自动重试一次；
- 高消耗操作显示预计处理范围；
- 不向日志写入 API Key；
- 默认不长期保存完整提示词和论文正文副本。

### 7.5 调用日志

记录以下脱敏信息：

- 模型名；
- 任务类型；
- 开始时间和结束时间；
- 调用耗时；
- 输入和输出 token（接口返回时）；
- 重试次数；
- 成功状态；
- 错误类型和脱敏错误信息。

## 8. 统一路由与 LangGraph

### 8.1 路由结构

```text
统一路由图
├─ 普通研究问答子图
├─ 文献发现子图
├─ 文献精读子图
└─ 选题与框架诊断子图
```

### 8.2 两级路由

1. 规则优先：上传 PDF、点击导入、选择论文、发起检索等明确操作直接进入对应子图；
2. 模型判断：自然语言意图模糊时，由 Qwen 识别模式和置信度。

回复界面显示当前模式，用户可以手动纠正。路由置信度不足时，系统先询问用户，不擅自进入高成本或不相关流程。

### 8.3 回答策略

- 普通知识问题直接回答；
- 文献发现任务先明确检索目标，再执行真实检索；
- 文献精读默认采用引导方式，也允许用户要求直接分析；
- 选题和研究设计任务先诊断关键信息缺口，用户也可跳过追问；
- 所有子图结束后更新项目档案并回到统一聊天入口。

### 8.4 共享状态

共享状态只保存流程所需字段：

- 项目 ID；
- 会话 ID；
- 当前模式和阶段；
- 最近消息和会话摘要；
- 当前选择的论文；
- 检索到的证据块；
- 待用户回答的问题；
- 结构化输出；
- 工具和模型错误；
- 回退状态。

## 9. 文献发现工作流

```text
用户输入研究兴趣或选题
→ 判断领域和检索目标
→ Qwen 生成英文检索词
→ LangChain 调用 arXiv
→ 获取最多 20 篇候选文献
→ 后端规则去重和过滤
→ Qwen 按相关性筛选 5–10 篇
→ 标注用途和推荐理由
→ 保存到项目
```

### 9.1 检索规则

- 支持按相关性排序；
- 支持按发布日期排序；
- 不展示 arXiv 未提供的可靠引用次数；
- 题录信息完全以 arXiv 返回结果为准；
- Qwen 不得自行补写不存在的文献；
- 每篇结果提供真实 arXiv 页面或 PDF 链接。

### 9.2 arXiv PDF 导入

- PDF 不超过 10 MB：下载并解析前 60 页；
- PDF 超过 10 MB：仅导入题录和摘要；
- 下载失败：保留题录和原文链接；
- 用户可以另行上传压缩后的 PDF。

## 10. PDF 解析与 OCR

### 10.1 输入限制

- 文件格式：PDF；
- 文件大小上限：10 MB；
- 允许上传超过 60 页的 PDF；
- 只解析、索引和分析前 60 页；
- 界面始终显示“仅分析前 60 页”。

### 10.2 解析流程

```text
PDF 导入
→ PyMuPDF 提取文本、页码、文本块和坐标
→ 检测文本数量、乱码比例和可读性
→ 文本质量不足时启动 Tesseract OCR
→ 识别基础章节结构
→ 按章节、段落和长度切分
→ 写入 PaperChunk 和 FTS5 索引
→ 更新解析任务状态
```

### 10.3 OCR 策略

- OCR 是自动兜底，不默认处理所有 PDF；
- 默认语言为 `chi_sim+eng`；
- 可以使用 `osd` 辅助方向检测；
- OCR 结果必须标记来源；
- OCR 失败不删除已成功提取的文本；
- 用户可以手动要求重新使用 OCR 解析；
- 不保证复杂公式、表格和双栏顺序完全准确。

## 11. 论文证据索引

首版使用轻量“论文证据索引”，不建设完整向量知识库：

```text
PaperChunk 数据
+ SQLite FTS5
+ Qwen 对候选证据进行相关性判断
```

每个文本块至少保存：

- 所属论文；
- 页码；
- 章节名称；
- 原文；
- 块序号；
- 页面坐标；
- 是否来自 OCR；
- 解析质量标记。

### 11.1 检索流程

```text
用户问题
→ 生成关键词或查询表达
→ FTS5 召回候选文本块
→ 按章节、页码和关键词规则重排
→ Qwen 从候选中选择有效证据
→ 基于证据生成回答
```

当关键词检索不足以支持后续效果时，再评估加入 FAISS 或嵌入检索，不在首版预先引入。

## 12. 文献精读

### 12.1 引导精读

采用：

```text
提问
→ 用户回答
→ 提示或纠偏
→ 下一问题
→ 形成文献卡片
```

Qwen 根据用户阅读目的、论文内容和已有回答动态决定问题，不使用机械固定顺序。每轮控制问题数量，避免一次性给新手过多任务。

### 12.2 快速分析

用户可以点击或提出“直接给我分析”，系统直接生成结构化分析。结果标记为“AI 辅助分析”，并附原文证据和页码。

### 12.3 文献卡片

卡片至少包含：

- 基本题录；
- 研究主题；
- 研究问题；
- 文献缺口；
- 理论框架；
- 核心概念；
- 变量或分析维度；
- 数据来源；
- 研究方法；
- 论证链；
- 可借鉴点；
- 迁移风险；
- 对用户选题的启发；
- 对应证据。

### 12.4 多篇论文对比

- 最多选择 3 篇；
- 比较研究问题、理论、方法、数据和结论；
- 每个关键比较项附页码证据；
- 只基于各论文已解析的前 60 页；
- 结果可编辑并导出 Markdown。

## 13. 选题与框架诊断

### 13.1 动态诊断方式

系统根据用户材料和项目上下文动态识别研究类型。量化、质性、案例、综述、工程实验等检查项可以组合使用，不要求用户只能选择一种类型。

可组合检查维度包括：

- 研究对象与边界；
- 研究问题；
- 概念定义；
- 理论依据；
- 研究方法；
- 数据或材料；
- 指标和评价方式；
- 论证链；
- 可完成性；
- 证据边界；
- 潜在风险。

### 13.2 文献参与诊断

诊断默认参考当前项目中已收藏或精读的论文，并明确列出使用的文献。输出区分：

- 文献证据支持的判断；
- 基于用户材料的合理推断；
- 尚需补充证据的内容；
- 高风险或无法确认的判断。

没有相关文献时，系统仍可基于用户材料给出初步诊断，但必须说明证据限制。

## 14. 项目记忆

### 14.1 对话优先、自动沉淀

- 用户无需先创建项目；
- 首次产生研究内容时自动创建未命名项目；
- 用户可以稍后修改项目名称和研究方向；
- 检索、精读和诊断成果写入当前项目；
- 用户可以切换和继续历史项目。

### 14.2 模型上下文

每次模型调用原则上只发送：

```text
最近 10–20 条消息
+ 自动维护的会话摘要
+ 当前项目结构化档案
+ 当前任务检索到的论文证据
```

全部原始消息完整保存在 SQLite，但不在每次调用中重复发送。

## 15. 数据模型

### 15.1 核心实体

| 实体 | 职责 |
|---|---|
| Project | 项目名称、研究主题和结构化研究档案 |
| Session | 项目内对话会话和会话摘要 |
| Message | 用户、助手和系统事件消息 |
| Paper | arXiv 或上传论文的题录、文件和解析状态 |
| PaperChunk | 原文片段、页码、章节、坐标及 OCR 标记 |
| Artifact | 文献卡片、诊断报告、对比表和研究框架 |
| Task | PDF 下载、解析、OCR 和索引任务 |
| ModelCallLog | 脱敏后的模型调用统计和错误 |
| Setting | 本地非敏感设置 |

### 15.2 Artifact 统一成果模型

首版不为每种成果创建独立业务表。`Artifact` 通过类型字段和 JSON 内容保存：

- 文献卡片；
- 选题诊断；
- 研究设计；
- 论文框架；
- 多论文对比；
- 检索和阅读路径。

这可以减少数据库复杂度，同时通过 Pydantic Schema 保证不同类型的内容结构。

### 15.3 文件存储

PDF、页面截图、解析中间文件和导出文件保存在项目本地 `data/` 目录。数据库使用相对路径，便于将整个项目文件夹复制到另一台 Windows 电脑。

## 16. 后台任务

PDF 下载、解析、OCR 和索引使用 FastAPI 本地后台任务，不引入 Celery。

任务状态：

```text
等待
→ 下载中（可选）
→ 解析中
→ OCR 中（可选）
→ 建立索引
→ 完成 / 失败 / 已取消
```

要求：

- 状态和进度写入 SQLite；
- 前端通过普通状态接口轮询；
- 支持用户取消尚未完成的任务；
- 程序重启后，将遗留的处理中任务标记为“已中断，可重试”；
- 单篇论文失败不影响项目内其他论文。

## 17. API 通信

### 17.1 对话流

对话回复使用 SSE，事件类型至少包括：

- 当前模式；
- 工作阶段；
- 文本 token；
- 检索摘要；
- 证据引用；
- 结构化成果；
- 完成；
- 错误。

SSE 连接中断时，已生成的消息和任务状态保存在 SQLite。首版不引入 WebSocket。

### 17.2 后台任务

PDF 解析进度通过普通 HTTP 状态接口查询，不与聊天 SSE 强绑定，以降低实现复杂度。

## 18. 本地设置

设置页允许配置：

- 百炼 API Key；
- 模型名；
- `base_url`；
- Tesseract 可执行文件路径；
- OCR 语言；
- 数据存储目录；
- 输入和输出 token 限制；
- PDF 页数和大小限制。

设置页提供：

- 模型连接测试；
- OCR 可用性测试；
- 数据目录可写性检查。

API Key 默认保存在本机 `.env`，不存入 SQLite，不在界面显示完整值，也不得提交到版本控制。

## 19. 错误处理与降级

| 故障 | 降级行为 |
|---|---|
| Qwen 不可用 | 保留 PDF 解析、arXiv 检索和历史成果查看 |
| arXiv 不可用 | 保留本地 PDF 精读、诊断和普通问答 |
| arXiv PDF 下载失败 | 保存题录和原文链接 |
| PDF 超过 10 MB | 本地上传拒绝；arXiv 导入退化为题录和摘要 |
| PDF 超过 60 页 | 接受文件，但只处理前 60 页并持续提示 |
| OCR 失败 | 使用已经提取的文本，标记无法识别页面 |
| 多模态失败 | 回退到文本分析，提示人工核验复杂内容 |
| JSON 校验失败 | 自动修复一次，仍失败则返回普通文本 |
| SSE 中断 | 保存已生成内容，允许重新读取或重新提交 |
| 程序重启 | 中断任务标记为可重试 |

错误信息应面向用户说明可执行的下一步，同时在日志中保留脱敏技术细节。

## 20. 安全与学术质量

- 不生成或补写不存在的参考文献；
- 不编造数据、访谈内容和实验结果；
- 不把模型推断伪装成论文原文结论；
- 对 OCR、摘要分析和全文分析进行明确区分；
- 只向百炼发送当前任务需要的文本片段或截图；
- 不在日志记录 API Key；
- 默认不长期保存完整模型提示词；
- 用户上传内容只在本机目录和百炼 API 调用范围内处理；
- 诊断结果明确区分证据、推断和待核实内容。

## 21. 运行与迁移

首版只保证 Windows 环境运行。开发与演示环境要求：

- Python 虚拟环境 `py39232`；
- Node.js；
- Tesseract OCR；
- 可访问阿里云百炼和 arXiv 的网络；
- 用户自行配置百炼 API Key。

最终交付应包含：

- Python 依赖清单；
- Node.js 版本和前端依赖；
- `.env.example`；
- Windows 安装与迁移说明；
- Tesseract 配置说明；
- 一键启动脚本；
- 数据目录说明。

SQLite、上传文件和解析结果全部采用项目内相对路径。用户将整个文件夹复制到另一台已配置 Python、Node.js 和 Tesseract 的 Windows 电脑后，可以继续运行。

## 22. 验收标准

课程演示版后端达到以下条件即视为完成：

1. 无需登录即可创建自动沉淀的研究项目；
2. 单一聊天入口可以正确路由四种工作模式，并允许用户纠正；
3. 普通研究问题可以流式回答；
4. arXiv 可以返回真实文献，并推荐 5–10 篇；
5. 可以导入 arXiv 论文或上传不超过 10 MB 的 PDF；
6. 可以解析前 60 页，并在必要时调用 Tesseract OCR；
7. 可以检索原文证据并返回真实页码；
8. 可以完成引导精读和快速分析；
9. 可以生成结构化文献卡片；
10. 可以比较最多三篇论文；
11. 可以结合项目文献动态诊断选题和框架；
12. 结构化输出可以通过 Pydantic 校验或合理降级；
13. 项目、消息、论文、任务和成果在程序重启后仍然存在；
14. 主要外部服务故障时，未受影响的功能仍可继续使用；
15. 文献卡片、诊断报告和对比表可以编辑并导出 Markdown。
